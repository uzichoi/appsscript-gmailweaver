-Goal-
Given a text document and a list of already-extracted entities, identify all valid relationships among those entities.
Each relationship must use exactly one of the predefined relationship types below.
Only output relationships that are clearly supported by the text and the provided entities.

-Relationship Types-
Use only the following relationship types:

- SENT_BY:
  Direction: EMAIL -> PERSON
  Meaning: the person is the sender of the email

- SENT_TO:
  Direction: EMAIL -> PERSON
  Meaning: the person is a direct recipient of the email

- CC_TO:
  Direction: EMAIL -> PERSON
  Meaning: the person is cc'd on the email

- DISCUSSED:
  Direction: EMAIL -> TOPIC or EMAIL -> EVENT
  Meaning: the email discusses the topic or mentions / explains the event as part of its content

- BELONGS_TO:
  Direction: PERSON -> ORGANIZATION or LABEL -> EMAIL
  Meaning:
  - PERSON -> ORGANIZATION: the person belongs to, works in, or is affiliated with the organization
  - LABEL -> EMAIL: the label is assigned to the email

- SCHEDULED_FOR:
  Direction: EVENT -> EMAIL
  Meaning: the event is scheduled, announced, confirmed, or coordinated through the email

- WAS_ATTACHED_TO:
  Direction: EMAIL -> ATTACHMENT-LIKE ENTITY
  Meaning: a file/document entity was attached to the email
  Note: only use this type if a corresponding attachment-like entity is present in the provided entity list

-Important Rules-
1. Use only the entities explicitly provided in {entities}. Do not create new entities.
2. Assign exactly one relationship type per output line.
3. If no predefined type fits clearly, skip the pair.
4. Do not infer weak or speculative relationships.
5. The same source-target pair may appear more than once only if each line uses a different valid relationship type and each is clearly supported by the text.
6. Respect direction strictly.
7. DISCUSSED is for email content about a Topic or Event. Do not use DISCUSSED for PERSON-ORGANIZATION, LABEL-EMAIL, or sender/recipient relations.
8. BELONGS_TO is only for:
   - PERSON -> ORGANIZATION
   - LABEL -> EMAIL
   Do not use it for any other pair.
9. SCHEDULED_FOR is only for EVENT -> EMAIL.
10. WAS_ATTACHED_TO is only valid when the attachment/file/document is already present as an extracted entity.
11. If the provided entity list contains no attachment-like entity, do not output WAS_ATTACHED_TO.
12. relationship_strength must be an integer from 1 to 10:
   - 9-10: explicit and direct in the text
   - 7-8: strongly supported but slightly indirect
   - 4-6: plausible but less central
   - 1-3: weak; usually skip unless still explicit

-Steps-
1. Review the entity list in {entities} and the source text.
2. Identify all source-target pairs that match one of the allowed relationship patterns above.
3. For each valid pair, extract:
   - source_entity
   - target_entity
   - relationship_type
   - relationship_description
   - relationship_strength
4. Write the relationship_description in Korean in one sentence, grounded in the text.
5. Return output in Korean as a single list. Use **{record_delimiter}** as the list delimiter.
6. When finished, output {completion_delimiter}

Format each relationship as:
("relationship"{tuple_delimiter}<source_entity>{tuple_delimiter}<target_entity>{tuple_delimiter}<relationship_type>{tuple_delimiter}<relationship_description>{tuple_delimiter}<relationship_strength>)
